# NeoOLAF Evaluation KG Projection

This patch adds a strict projection step from:

```text
kg_local.json
kg_inferred.json
```

to:

```text
evaluation_kg_local.json
evaluation_kg_inferred.json
evaluation_kg_combined.json
evaluation_projection_report.json
```

The source/gold file is used only to define the entity universe and allowed
relation vocabulary. Gold triples are not copied as predictions.

The `entities` array in each evaluation KG contains only source-aligned entities
that actually occur in NeoOLAF's native KG triples. The `relations` array
contains only predicates used by accepted projected triples. The projector does
not copy the complete gold entity set or complete allowed relation list into
the prediction.

## Recommended DocRED Command

```bash
python evaluation_kg_projection.py \
  --kg-local-json runs/.../data/exports/kg_local.json \
  --kg-inferred-json runs/.../data/exports/kg_inferred.json \
  --source-doc-json data/RAGTreeDatasets/docred/dev.jsonl \
  --document-id "DocRED - e37288ca6012859f" \
  --ontology-path data/ontology/redocred_ontology.ttl \
  --relation-vocab-json data/RAGTreeDatasets/docred/relation_vocab.json \
  --relation-filter-mode list-only \
  --output-dir runs/.../data/exports
```

The ontology argument must point to the fixed dataset/reference ontology. Do
not pass an ontology generated by the current NeoOLAF run, because that would
make newly discovered relations automatically eligible for evaluation.

## Relation Filter Modes

```text
ontology-only  accept only RDF/OWL properties in the reference ontology
list-only      accept only relations in the explicit/source-derived list
union          accept relations found in either vocabulary
intersection   accept only relations represented in both vocabularies
```

For DocRED, `list-only` is recommended with the official DocRED relation
vocabulary. Alternatively, pass the complete source dataset through
`--source-relation-vocab-json`; only relation IDs and labels are collected, not
gold subject-object pairs.

## Why Relation Vocabulary Is Optional

In principle, the allowed relations should come from the ontology. In practice,
some Re-DocRED ontology files contain mostly classes and subclasses, while the
benchmark still needs Wikidata-style relation IDs such as `P19`, `P27`, `P131`,
or labels such as `place of birth`.

So the available fixed vocabulary inputs are:

```text
--ontology-path
--relation-vocab-json
--source-relation-vocab-json
```

NeoOLAF remains free to add relations to its own generated ontology and native
KG exports. The restriction applies only when creating `evaluation_kg*.json`.

## Wikipedia-Free Benchmark Run

The replacement notebook defaults to:

```python
DISABLE_WIKIPEDIA = True
```

It adds `experiments/methods/neoolaf_offline_runtime` to the child process
`PYTHONPATH`. Python then loads `sitecustomize.py`, which intercepts only
Wikipedia/Wikimedia requests and immediately returns an empty MediaWiki result.
NeoOLAF source code is not modified, and OpenRouter requests are not blocked.

This avoids Wikipedia rate-limit waits and prevents external entity enrichment
from contaminating fixed-dataset DocRED evaluation.

## Strict Behavior

The projector accepts a triple only if:

```text
subject -> source entity
object -> source entity
predicate -> allowed relation
```

A broken KG triple like:

```json
{
  "subject": {"label": "Jack Gantos", "type": "relation"},
  "predicate": {"label": "Jack Gantos"},
  "object": {"label": "Westmoreland County", "type": "entity"}
}
```

will be rejected because `Jack Gantos` is not an allowed predicate. The report
will contain:

```text
invalid_predicate_relation
predicate_is_entity_label
```

The subject role can be repaired from the source entity universe, but the
predicate cannot be silently repaired from the gold answer.

## Runner Integration

After each document run, call this projector once the native KG exports exist.
Then print:

```text
[NeoOLAF benchmark][exports] run_dir=...
[NeoOLAF benchmark][exports] exports_dir=.../data/exports
[NeoOLAF benchmark][exports] kg_local=.../kg_local.json
[NeoOLAF benchmark][exports] kg_inferred=.../kg_inferred.json
[NeoOLAF benchmark][exports] evaluation_kg_local=.../evaluation_kg_local.json
[NeoOLAF benchmark][exports] evaluation_kg_inferred=.../evaluation_kg_inferred.json
[NeoOLAF benchmark][exports] evaluation_kg_combined=.../evaluation_kg_combined.json
```

At dataset level, also append these paths to an artifact index JSONL.
